# Yonder 工程规约

本文件是 yonder 项目的长期强制规约，适用于所有贡献者、自动化工具和编码代理。后续经项目所有者确认的新要求必须补充到本文件。除非项目所有者明确修改，以下规则不得弱化、绕过或以“临时方案”为由违反。

## 决策与审批

- 新增、删除、替换、升级任何直接依赖之前，必须先向项目所有者提交方案并获得明确确认。
- 实现任何功能或作出架构、数据模型、并发模型、持久化、协议、错误处理、性能策略等设计决策之前，必须先说明设计思路、备选方案和取舍，并获得明确确认。
- 发现需求、约束、术语、验收标准或实现边界存在任何不明确、冲突或疑问时，必须先沟通确认，禁止基于猜测直接实现。
- 未经确认，不得扩大任务范围，不得进行顺手重构，也不得引入新的工程约定。
- 已确认的设计如果在实现中证明不可行，必须停止相关实现并重新申请确认，不得静默改用其他方案。

## Rust 安全红线

- 所有第一方 Rust 代码必须是 safe Rust，禁止出现 `unsafe` 块、`unsafe fn`、`unsafe trait`、`unsafe impl`、不安全 FFI 或任何规避 `unsafe_code = "forbid"` 的做法。
- 禁止为 `Send` 或 `Sync` 编写不安全实现；线程安全应由所有权、生命周期、类型系统和自动 trait 推导保证。
- 第一方代码必须保证内存安全、线程安全且无数据竞争。涉及并发时，设计文档必须明确共享状态、同步边界、取消、关闭、背压和故障传播语义。
- 第三方依赖内部的 `unsafe` 不受第一方源码禁令约束，但引入前必须审查其安全记录、已知 soundness 问题、维护状态和不安全代码风险；存在未解决 soundness 问题的依赖默认不得引入。
- 禁止通过关闭 lint、局部 `allow`、生成代码、构建脚本或宏等方式绕过安全红线。

## 依赖策略

- 对已有成熟、流行且维护活跃的 crate 能可靠解决的问题，优先使用社区实现，尽量减少第一方手写代码；禁止重复实现成熟的算法、协议、解析器、数据结构、并发原语或平台抽象。
- 所有直接依赖默认必须选用经确认时的全局最新稳定版本。版本信息必须通过 crate 官方资料或 crates.io 核实，禁止凭记忆填写。
- 如果已确认采用的最新稳定上层 crate 在公开 API 中绑定了不兼容的旧主版本，则允许直接依赖该兼容主版本的最新稳定补丁版。此例外必须逐项说明原因、证明无法直接使用全局最新版、确认不存在未解决的安全公告，并获得项目所有者明确批准；上游发布兼容稳定版本后必须立即重新评估。禁止借此使用任意旧版本或规避升级。
- 每个直接依赖都必须显式设置 `default-features = false`，并仅显式开启实际需要且经过确认的 features。
- 依赖提案必须说明：用途、版本、成熟度与维护状态、替代方案、选择理由、许可证、安全风险、MSRV 影响、二进制体积影响、性能与分配影响、静态链接及跨平台影响，以及准备启用的完整 feature 列表。
- `Cargo.lock` 必须纳入版本控制。依赖升级必须经过与新增依赖相同的评审、测试和确认流程。
- 标准库可以充分满足且不会导致重复造轮子时，优先避免额外依赖；若标准库方案与成熟 crate 方案存在取舍，仍须先提交比较并确认。

## 架构与类型建模

- 功能设计必须面向 trait。可替换行为、外部能力和模块边界必须由职责单一、对象安全需求明确、可测试的 trait 表达，使实现或 crate 能在不影响调用方的情况下替换。
- 无替换语义的纯函数、简单 newtype 方法和模块内部实现细节不强制包装成 trait；禁止为了形式上的“面向 trait”制造无意义抽象。
- trait 应保持最小能力面，不得泄漏具体实现类型；是否使用静态分发或动态分发必须结合零成本目标说明并确认。
- 禁止 primitive obsession。领域概念必须使用具有明确不变量的 newtype、枚举、状态类型或其他类型安全模型表达，不能让无语义的 `String`、整数、布尔值在模块边界扩散。
- 非法状态应尽可能做到无法构造；解析、验证和状态转换必须在边界完成，内部代码只处理已验证类型。
- 错误必须使用结构化、类型安全且可组合的模型。禁止用裸 `String` 充当跨边界错误类型，禁止吞错、静默降级或使用 panic 处理可恢复错误。
- 同时禁止过度设计和简陋设计。抽象必须有明确替换点、测试价值或复杂度收益，不得建立没有实际语义的层级。
- 保持实现简洁是项目的明确目标：在满足安全、正确性、性能、可维护性和可替换性要求的前提下，优先选择定制代码更少、状态更少、控制流更直接且维护面更小的方案。
- 简洁不能作为省略必要防护、验证、测试或错误处理的理由。任何以简化实现为目的的取舍，都必须事先评估其对安全边界、正确性、性能、资源、跨平台兼容性和后续演进的影响，明确残余风险并获得项目所有者确认。

## 性能与资源

- 零成本抽象和零分配是优先设计目标。热路径不得出现未经证明必要的堆分配、复制、动态分发、锁竞争、阻塞或无界资源增长。
- 无法做到零分配时，必须量化分配次数、大小和生命周期，说明原因和替代方案，并在实现前获得确认。
- 性能结论必须由可复现的 benchmark、profiling 或资源测量支持，禁止仅凭直觉声称“高性能”或“零成本”。
- 发布产物以运行性能为第一优化目标，同时持续测量吞吐、延迟、启动速度、峰值内存和二进制体积，并为体积设置可验证的约束。后续改变优化优先级仍须先确认。
- `dev` 和 `test` profile 优先开发反馈速度与测试可靠性；`release` profile 使用 `opt-level=3`、fat LTO、单 codegen unit、符号裁剪和 `panic=abort`。profile 变更必须用 benchmark 和产物测量验证。

## 测试与质量门禁

- 所有实现必须真实、完整、可靠。禁止提交占位实现、伪实现、生产路径 mock、`todo!()`、`unimplemented!()` 或掩盖失败的 fallback。
- 每个功能都必须有与风险匹配的单元测试、集成测试、端到端测试、属性测试、模糊测试、性能测试和必要的平台测试。若某类测试确实不适用，必须说明理由并获得确认，不能自行省略。
- 覆盖率采用项目所有者已确认的风险分级门禁：每个可工作的原生目标必须独立达到聚合 line `>=95%`、function `>=95%`、region `>=90%`、每文件 line `>=75%`、工具可测 branch `>=75%`。不得跨平台合并掩盖单一平台缺口；门槛只能在证据支持时提高，任何降低仍须重新审批。
- 已批准仅在 nightly coverage 专用构建中使用 `#[coverage(off)]` 排除 `#[cfg(test)]` 单元测试模块；独立 integration test、example 与 benchmark harness 使用 `cargo-llvm-cov 0.8.7` 官方默认的路径排除规则，fuzz harness 不属于生产 workspace coverage 输入。这些夹具不属于产品可执行逻辑，且其中“若执行即失败”的保护闭包不应为了覆盖率被人为调用。任何排除不得应用于生产模块、生产错误分支、平台实现、CLI、协议解析器、测试 seam 的生产实现或由生产源码映射出的可测分支。stable、MSRV、Miri、sanitizer 和正常发布构建不得启用 coverage 专用 cfg。
- 已确认 `aarch64-pc-windows-msvc` 的覆盖率临时例外：Rust 上游 issue `#150123` 修复前，官方 LLVM coverage 无法在该目标可靠写出 profraw，因此该目标继续执行原生 Clippy、测试、release 构建、PE import/static CRT 检查和空目录 smoke，但不计入 coverage job。其余五个可工作的原生发布目标分别执行上述风险分级门禁；上游修复后必须立即取消例外并补齐第六目标。
- 覆盖率未达门禁时必须报告精确数据和缺口分类，不得把失败结果称为完成、通过发布门禁或以排除替代真实生产路径测试。即使数值超过门槛，安全、认证、协议、状态机、资源边界和并发故障路径仍必须按风险补足定向、属性、模糊、压力与端到端证据，覆盖率百分比不能替代正确性证明。
- 并发功能必须验证竞态、死锁、取消、超时、乱序和故障路径；解析器和外部输入边界必须加入持续模糊测试。
- 性能测试必须定义基线和允许的回归阈值；没有测量数据不得宣称满足性能、内存或零分配目标。
- 最低通用门禁包括格式检查、所有 target 的 Clippy 零警告、完整测试、覆盖率、文档测试、依赖审计、许可证检查、静态链接检查和支持平台 smoke test。引入相应工具或依赖前仍须走审批流程。
- 修复缺陷时必须先添加能够复现问题的回归测试，再提交修复。
- 已批准 CI/release 工具固定为 `cargo-llvm-cov 0.8.7`、`cargo-cyclonedx 0.5.9` 和 `actionlint 1.7.12`；升级继续遵守工具与依赖审批流程。全部 GitHub Actions 必须固定到完整 commit SHA，禁止使用可移动 tag。
- 发布必须为每个 target 分别产出 `yon` 与 `yon-relay` 的单文件归档：Unix 使用 `.tar.gz`，Windows 使用 `.zip`，每个归档恰好包含一个规范名称的可执行文件。Release 同时发布 CycloneDX 1.5 JSON SBOM、`Cargo.lock`、全资产 SHA-256 和构建 provenance。

## 实施流程

1. 明确需求、非目标、平台范围、可观测行为和量化验收标准。
2. 提交 trait 边界、类型模型、错误模型、资源与并发模型、依赖候选、测试矩阵和性能计划。
3. 等待项目所有者明确确认设计及每个依赖。
4. 进行最小范围实现，并在开发过程中持续运行相关门禁。
5. 完成全部测试、静态链接及跨平台验证，提供可复现结果和残余风险。
6. 若任一门禁未通过或无法真实验证，不得声称完成。

## 0.1.0 用户体验门禁

- 实际使用体验是 `0.1.0` 的硬性验收目标，不能以协议正确、测试通过或功能可达替代。host 与 connect 的网络准备、重试、回退、认证和终端启动必须持续提供非敏感、可理解且不会污染远端终端的状态反馈。
- 当 stdout 与 stderr 共享终端画面时，远端终端 Active 期间禁止 tracing 或第三方诊断事件写入该画面；详细诊断必须显式重定向。应用错误、终端恢复失败和退出码映射等必要结束期报告必须在远端输出结束并另起一行后显示，不能被日志过滤静默吞掉。
- 进度反馈必须在网络活动前同步出现，长等待期间最长 `1s` 必须有一次可见心跳；提示必须限制在当前终端单行宽度内，收到 `TerminalReady` 后先完整清除再转发远端输出。`TERM=dumb`、非终端或无法可靠取得终端尺寸时禁止发送光标控制序列。
- 有可用 relay 候选时，controller 的 DCUtR 优先窗口固定为 `3s`；真实网络矩阵的 controller 完整终端会话必须执行明确的时延上限，不得只记录数值而不判定。Windows ConPTY 与 Unix PTY 的默认交互路径都必须有原生端到端回归证据。
- 直连优先是路径选择的硬语义：优先窗口内出现已完成 libp2p 传输身份认证的直连候选时，controller 必须只在直连集合内部按有界质量证据选择；relay RTT 不得覆盖可用直连。`libp2p-dcutr` 成功事件中的 `ConnectionId` 只标识本端发起的物理连接，不能假定两端事件指向同一物理连接。controller 是唯一选路者并按本地 `ConnectionId` 关闭 loser；host 只在名册收敛后绑定剩余的唯一认证连接。真实网络测试必须断言最后一次选路记录，而不是日志中曾经出现过的路径。

## Design Freeze v1

- `docs/design/` 是已经由项目所有者整体确认的 v1 产品、架构、协议、安全、依赖、验证与追踪基线；实现必须逐项遵守。
- 冻结文档与本规约共同构成实现输入。发现冲突、不可行或需要改变可观测行为时必须暂停受影响实现、说明证据和取舍，并在没有既有明确授权时重新审批；任何情况下都不得静默取舍。
- 冻结中列出的全部直接依赖、精确版本、`default-features = false`、显式 feature、MSRV、许可证策略和三个精确上游例外均已批准；任何变化继续遵守本文件的依赖审批流程。
- v1 实现任务必须能映射到 `docs/design/traceability.md` 的需求 ID、责任 package 和可执行证据；无法映射视为范围扩大。
- 冻结是可追踪的受控设计基线，不是产品目标本身。项目所有者已明确要求：真实网络、平台、运维、功能或用户体验证据证明冻结项不合理时，以生产可用的远程终端目标为先，必须纠正设计，禁止为了形式上的冻结一致性保留已知问题。
- 上述纠正不得静默进行：设计、实现、测试、验证记录和追踪矩阵必须同步更新，并说明证据与取舍；不得借此降低安全、正确性、测试或发布门禁。新增、替换或升级直接依赖仍必须单独获得项目所有者明确确认。

## 已确认的连接码设计

- 主控端和被控端必须预先配置并连接同一个自建中继；连接码不携带中继地址、中继身份或传输协议信息。
- 中继只保存内存中的临时注册映射，不建立持久连接码数据库。正常注销成功后映射立即失效；异常退出与暂时断网在分布式环境中不可区分，必须按下述断线宽限规则处理。
- 连接码使用 Crockford Base32 的数字与大写字母字符表 `0123456789ABCDEFGHJKMNPQRSTVWXYZ`，规范显示为四组四位：`XXXX-XXXX-XXXX-XXXX`。
- 连接码的 16 个有效字符由四字符公开定位码和十二字符端到端认证秘密组成；连字符仅用于显示和输入分组，不承载信息。
- 规范输出仅使用大写字符并避开 `I`、`L`、`O`、`U`；输入接受大小写、规范分组或无连字符形式，`O/o` 映射为 `0`、`I/i/L/l` 映射为 `1`，拒绝 `U/u`、空白和其他分隔符。
- 连接码每次启动重新生成，仅在该被控端进程存活期间有效，并且只允许一个主控端成功使用。网络连接、打洞或认证失败不得消耗连接码；认证且终端会话建立成功后立即消耗。
- 第一组四字符是 20 bit 公开定位码，由中继原子分配并临时映射到被控端连接；定位码不承担认证或保密职责。
- 后三组十二字符必须由被控端使用密码学安全随机数生成器均匀生成，形成 60 bit PAKE 秘密；中继不得接收或保存该秘密。
- 主控端通过公开定位码找到被控端后，两个端点必须使用标准 PAKE 完成相互认证和密钥确认，并将认证结果绑定到当前已认证的端到端 libp2p 连接及双方身份；该要求必须统一覆盖使用 Noise 的 TCP/WS 和使用 TLS 1.3 的 QUIC。
- 不采用“从完整连接码经内存困难 KDF 派生 RendezvousId”的注册方案，避免产生确定性的离线猜码验证目标及不必要的启动内存开销。
- PAKE 采用 RFC 9807 OPAQUE：主控端是 OPAQUE Client，被控端是 OPAQUE Server。被控端启动时在本地内存中完成临时注册，只保留本次进程所需的 `ServerSetup` 与密码记录；原始 PAKE 秘密在输出连接码并完成注册后必须尽快清除。
- OPAQUE 密码套件固定采用 Ristretto255、TripleDH、SHA-512 和 Argon2id；认证结果必须绑定到当前 transport 已认证的双方 PeerId、双方 nonce 与唯一端到端连接上下文，统一覆盖 Noise 和 QUIC/TLS。
- 已批准直接依赖 `opaque-ke 4.0.1`，必须设置 `default-features = false`，仅启用 `argon2`、`ristretto255`；禁止启用默认 `serde`，因为认证状态只存在内存中。
- 已批准兼容直接依赖 `rand 0.8.7`，必须设置 `default-features = false`，仅启用 `getrandom`，用于满足 `opaque-ke` 暴露的 Rand 0.8 RNG trait 并调用操作系统 CSPRNG。全局最新版 Rand 0.10 无法满足 `opaque-ke 4.0.1` 的 Rand 0.8 公开类型约束，因此采用兼容主版本的最新稳定补丁；`0.8.7` 同时修复了 `0.8.6` 中恶意反序列化 `UniformChar` 可能违反内存安全的问题，禁止回退到 `0.8.6` 或更早版本。
- 已批准兼容直接依赖 `sha2 0.10.9`，必须设置 `default-features = false`，用于满足 `opaque-ke 4.0.1` 暴露的 Digest 0.10 类型约束并提供 SHA-512。
- OPAQUE 依赖树必须锁定已修复已知计时漏洞的 `curve25519-dalek >= 4.1.3, < 5`；实现前及每次升级时仍须运行完整依赖、安全和许可证审计。
- 已确认直接使用 `opaque-ke` 提供的 Argon2 KSF，以保持实现简洁；不为清除 Argon2 工作区而增加自定义 KSF 适配层或仅服务于该适配层的直接依赖。
- 已接受以下明确的残余风险：当前上游 Argon2 实现释放约 19 MiB 工作缓冲区前不保证清除其全部内容，因此在攻击者已经能够读取本进程内存、调试或转储进程，或取得 core dump、swap、休眠镜像的场景中，可能残留可用于本地内存取证的 Argon2 中间状态。该取舍不降低 safe Rust 的内存安全与数据竞争保证，也不改变正常网络攻击模型下 OPAQUE 的认证和端到端保密属性；具有上述本地访问能力的攻击者本就能够接触该进程的 OPAQUE 状态、终端数据和当前用户权限。
- 上述接受仅限 Argon2 大型工作缓冲区。原始十二字符 PAKE 秘密、最终会话密钥及其他由第一方明确持有的小型秘密仍必须在各自生命周期结束时使用经批准的可靠机制尽快清除。若上游提供简单可靠的完整工作区清除能力、威胁模型扩大，或审计发现风险判断发生变化，必须重新评估本取舍。
- OPAQUE 的 Argon2id 参数固定为版本 `0x13`、`m = 19456 KiB`、`t = 2`、`p = 1`。协议不协商、不动态调整这些参数，避免实现复杂度、降级空间和由对端控制的内存消耗；被控端启动时的本地注册和主控端登录必须使用完全相同的固定参数。
- 每个连接码同时最多允许一个 OPAQUE 认证交换，不建立等待队列；额外请求必须立即以统一的可重试结果拒绝。
- 每个连接码使用被控端本地令牌桶限制认证启动：容量为 `4`，每秒恢复 `1` 个令牌。仅在接收并通过结构校验的 OPAQUE `KE1` 后消耗令牌；因并发占用或令牌不足而拒绝的请求不消耗令牌，也不算一次认证失败。
- 单条 OPAQUE 认证消息的等待上限为 `10s`，完整认证交换的总上限为 `20s`。连接中断、消息非法、超时、未收到 `KE3` 或 `ServerFinish` 失败都必须作为认证失败处理，立即销毁本次临时状态并释放并发槽。
- 认证失败不得消耗连接码，不设置永久锁定，不使用递增退避；认证成功且终端会话建立后，必须以原子状态转换消耗连接码。已冻结的中继资源限制不能替代被控端本地认证保护。
- OPAQUE 必须在所选 libp2p 连接的专用认证子流上运行。被控端本地注册及登录时使用其 PeerId 作为 OPAQUE `server identifier`；注册时尚不知道主控端身份，因此 OPAQUE `client identifier` 留空。
- 每次认证的 OPAQUE `context` 必须按角色固定顺序包含：版本化协议域分离标识、20 bit 公开定位码、主控端 PeerId、被控端 PeerId、主控端生成的 32 字节随机数和被控端生成的 32 字节随机数。PeerId 必须取自当前已完成传输层身份认证的 libp2p 连接，禁止信任中继或对端在应用消息中另行声明的身份。
- 由于 `libp2p-stream 0.4.0-alpha` 的子流 API 不暴露 `ConnectionId`，认证采用已验证的唯一连接屏障：双方只有在目标 PeerId 的本地连接名册恰好包含一个 `ConnectionId` 时才开始认证，并把该唯一值记录为授权连接；败选连接必须先关闭。认证开始后出现第二条同 PeerId 连接必须撤销未提交授权并关闭全部相关连接，Active 后出现则终止已提交会话且连接码保持消费。`ConnectionId` 只用于本地名册，不进入 OPAQUE `context`。
- 不把 TCP、QUIC、WS、直连或中继路径类型纳入认证上下文，不依赖当前 rust-libp2p 未公开的 Noise/TLS exporter，也不为本绑定额外实现 HMAC 或应用层双重加密。OPAQUE 完成认证和上下文完整性保护，libp2p 的 Noise 或 QUIC/TLS 负责传输加密；OPAQUE 会话密钥在授权状态转换完成后必须尽快清除。
- 认证上下文允许在每次认证的冷路径上构造一个最大 `256 B` 的有界缓冲区，不得进入终端数据热路径。该有界分配是为保持实现清晰而接受的取舍，仍须通过分配测量验证；不为消除它引入额外依赖或复杂的定长栈编码器。若 rust-libp2p 后续提供跨传输、稳定且公开的 channel exporter，必须通过既定抽象重新评估。
- OPAQUE 认证子协议的首个线协议 ID 固定为 `/yonder/auth/1.0.0`，由 libp2p `StreamProtocol`/multistream-select 完成精确协商，消息体不重复携带版本字段；该完整协议 ID 同时作为 OPAQUE `context` 的域分离标识。
- 认证子流按固定顺序交换：主控端发送“32 字节主控随机数 + OPAQUE `KE1`”；被控端返回“`Proceed` + 32 字节被控随机数 + OPAQUE `KE2`”或统一 `Retry`；主控端发送 OPAQUE `KE3`；被控端成功执行 `ServerFinish` 并记录名册中的唯一授权连接后返回单字节 `Authenticated`。主控端收到 `Authenticated` 前不得建立终端子流；精确 tag、长度和等待值以 `docs/design/protocol.md` 为准。
- OPAQUE 三类线消息必须直接使用 `opaque-ke` 在固定密码套件下提供的定长 `serialize()`/`deserialize()`；保持 `serde` feature 禁用，不为认证消息引入通用序列化或 codec crate。固定长度字段使用精确读写，不增加长度前缀，不允许由输入声明分配大小。
- 未知状态、截断消息、非法 OPAQUE 数据及认证失败必须关闭认证子流，网络响应不得区分密码错误、协议错误或其他认证细节。`Retry` 只表达并发槽或令牌暂不可用，不算认证失败，也不消耗连接码。
- 未来不兼容的认证线格式必须使用新的协议 ID。只有协议协商明确不支持当前版本时才可按另行确认的兼容策略选择其他已批准版本；认证失败后禁止自动降级重试。
- 被控端必须用单一所有者持有的类型安全状态机管理连接码和会话生命周期，禁止用可产生矛盾组合的多个布尔值或跨任务共享锁表达。状态顺序为 `Advertised -> Authenticating -> AwaitingTerminal -> StartingTerminal -> Active -> Spent`。
- `Authenticating` 中认证失败、超时、唯一连接失效或额外同 PeerId 连接到达必须返回 `Advertised`。OPAQUE 成功后进入 `AwaitingTerminal`，授权只保留给名册中的唯一连接；其他认证请求返回 `Retry`。主控端必须在收到 `Authenticated` 后 `10s` 内打开两条终端子流且名册始终唯一，否则撤销授权并返回 `Advertised`。
- 收到合法的 terminal-control 与 terminal-data 两条子流后进入 `StartingTerminal`。PTY 或当前用户 shell 创建失败、连接断开，或 `TerminalReady` 写入/刷新失败时，必须终止已创建的子进程、清理全部资源、撤销授权并返回 `Advertised`，连接码不得消耗。
- PTY 和 shell 创建成功且 `TerminalReady` 已在终端子流上成功写入并刷新后，状态必须原子转换为 `Active`，该转换是连接码的权威消耗点。不得等待 shell 提示符或启动脚本完成；这些行为属于已经建立的终端会话。
- `Active` 后即使立即断线或会话结束也不得恢复连接码；进程若继续存活则进入 `Spent`。进程退出会销毁任意状态。
- 已接受以下可用性残余风险：传输刷新成功不能证明主控应用层已经读取 `TerminalReady`，因此连接在提交点后立即断开时，连接码可能已经消耗而主控未显示终端。额外的客户端确认只能移动而不能消除该不确定窗口，并会增加 RTT 与状态复杂度，当前不采用。
- 中继注册按被控端进程的临时 PeerId 归属。只有同一 PeerId 同时持有有效 Circuit Relay v2 reservation 且至少保有一条到中继的连接，定位码才处于 `Active`；任一条件失效都进入 `Suspended` 并保留 `120s`。即使连接仍在，reservation 到期也必须 Suspended；恢复两个条件后无感回到 Active。
- 正常注销成功时立即删除映射，不进入宽限期；`Suspended` 超过 `120s` 后必须删除并释放定位码。宽限期限使用单调时钟计算，宽限记录必须计入中继总体资源上限。
- 中继重启会丢失全部内存映射。被控端重连时应优先申请原定位码；若该定位码已经被占用，则必须重新生成完整的 16 位连接码及对应 OPAQUE 状态并明确输出新码，禁止只替换公开定位码而复用旧秘密。
- 已接受以下可用性边界：纯内存中继无法保证重启后保留原连接码，也无法保证异常退出时立即释放映射；若未来要求跨中继重启绝对保持连接码，必须重新评审持久化设计及其安全边界。

## 已确认的中继信任边界

- 中继必须按不可信基础设施设计；该约束同样适用于用户自建的中继。
- 中继可以获知连接所必需的元数据，包括双方网络地址、PeerId、连接时间、连接持续时间和转发流量大小；当前设计不承诺隐藏这些元数据。
- 中继可以拒绝服务、中断连接、延迟或丢弃数据；端点必须正确检测并报告故障，但无法从协议上保证恶意中继提供可用服务。
- 终端内容、按键、控制消息和后续文件数据必须在两个 `yon` 端点之间完成端到端加密与认证；中继只能注册、协调或转发密文。
- 中继不得能够读取或修改受保护内容，不得伪造、重放命令，不得冒充任意端点或劫持已认证会话。
- 中继不得通过正常协议直接获得原始连接码；注册、查询、端到端认证和唯一连接绑定的完整协议已冻结在 `docs/design/protocol.md` 与 `docs/design/architecture.md`。

## 已确认的中继注册容量与滥用控制

- 被控端只有先取得当前中继的有效 Circuit Relay v2 reservation，才允许注册连接码定位映射；注册映射不得替代或绕过 libp2p relay 自身的资源限制。
- 中继默认最多保存 `128` 个注册映射，`Active` 与 `Suspended` 状态共同计入该上限；该值允许由中继配置调整，但配置必须经过类型安全校验并同步评估连接数、内存和文件描述符资源。
- 每个被控端临时 PeerId 最多拥有一个注册映射；重复注册必须是幂等操作，不得额外占用定位码或容量。
- 每个来源网络前缀默认最多同时存活 `32` 个由该前缀首次创建的注册映射，IPv4 按 `/32`、IPv6 按 `/64` 归一化计数。创建前缀在映射生命周期内不变；Active 与 Suspended 都计入，Release 或宽限到期删除时递减。同一 PeerId 的幂等注册及 `120s` 宽限期内从其他网络恢复已有映射不属于新建、不转移计数，也不受新前缀配额阻断。
- libp2p relay 默认同步配置 `max_reservations = 128`，产品有效语义为每 PeerId 最多 `1` 个 reservation，reservation 有效期采用 `1h`；正常续租不得改变连接码或重新分配注册映射。锁定的 `libp2p-relay 0.21.1` 对 reservation/circuit 的 per-peer 上限使用 `current > configured` 判断，因此适配层对应字段必须配置为 `0` 才能得到有效上限 `1`；升级上游时必须以真实双连接 reservation 回归重新核对并移除不再需要的兼容值。若操作员调整总注册容量，relay reservation 容量必须一并校验，避免两层上限静默失配。
- reservation 建立速率优先复用 libp2p relay 原生的按 PeerId 与按 IP 限制，不为注册路径另行引入限流 crate。Yonder 自定义注册层只维护总容量、每 PeerId 唯一性和来源前缀新建计数。
- 注册容量或来源前缀配额已满时，不得驱逐、抢占或随机替换已有映射；必须拒绝新注册并断开相应未注册被控端连接，使其 reservation 得以及时释放。
- 注册表由中继网络状态机的单一所有者独占维护，使用有界容器与计数器，不为该状态增加跨任务共享锁。默认 `128` 映射下，第一方注册索引和状态的内存目标不超过 `256 KiB`，必须通过可复现测量验证。
- 上述限制用于降低单一来源和低成本 PeerId 轮换造成的资源占用，但不承诺抵御大规模分布式拒绝服务；中继可用性仍属于已确认的不可信基础设施边界。

## 已确认的定位查询语义

- 主控端向中继查询连接目标时，只允许发送连接码第一组对应的 20 bit 公开定位码；后三组十二字符 OPAQUE 秘密不得发送给中继，也不得进入中继日志、指标或错误信息。
- 中继查询协议只允许返回三类结构化结果：`Resolved`、`Retry` 与 `Unavailable`。不得用裸字符串或任意状态码扩展泄漏内部状态。
- `Resolved` 只在定位映射当前可用时返回，并且只携带建立 Circuit Relay v2 路径所必需的被控端 PeerId；不得附带被控端地址列表、连接码秘密、OPAQUE 状态或其他注册表字段。主控端使用已配置的同一中继与该 PeerId 建立 relay circuit，随后再按已确认的统一路径策略尝试 DCUtR 直连升级。
- `Retry` 用于定位映射处于 `Suspended` 或查询暂时受到资源限制的情况，并携带经过上限约束的重试等待时间；不得向查询方进一步区分断线、限速或内部资源原因。
- `Unavailable` 用于定位码不存在、已过期或已经消费的情况；网络响应和 CLI 均不得进一步区分这些原因，CLI 统一表达为连接码无效或已失效。
- 定位查询不得锁定、预占或消耗连接码，也不得改变被控端的认证或会话状态。`Resolved` 只表示发现了候选端点，最终授权仍必须由两个端点间的 OPAQUE 认证和精确连接绑定决定。
- 不为隐藏定位映射状态而人为加入随机响应延迟；该做法不能阻止成功结果暴露有效定位码，却会增加延迟、资源占用和协议状态。公开定位码可被尝试枚举的风险必须通过独立的查询速率与资源限制控制。

## 已确认的定位查询限速

- 受 `libp2p-stream` 只暴露 PeerId 的 API 约束，每个查询 PeerId 同时最多处理一个定位查询，并且 relay 本地该 PeerId 的连接名册必须恰好一条，来源前缀从该唯一连接取得；不建立等待队列，额外并发或名册不唯一必须立即返回统一的 `Retry`。
- 中继定位查询的默认全局限额为突发 `128` 次、持续恢复 `4 次/秒`；每个来源网络前缀的默认限额为突发 `32` 次、持续恢复 `1 次/秒`，来源前缀继续统一按 IPv4 `/32` 与 IPv6 `/64` 归一化。
- 查询处理顺序固定为：校验固定长度消息、检查全局限额、检查来源前缀限额、查询注册表。全局限额未通过时不得创建来源限速状态，防止随机地址或前缀制造无界状态。
- 完整但编码非法的查询必须消耗全局查询配额；截断、超长或违反线协议的消息必须直接关闭相应子流，不得进入注册表查询。
- 来源前缀限速状态空闲 `10min` 后清除，状态表硬上限为 `4096` 项。达到硬上限且清理后仍无空间时，新来源查询必须返回 `Retry`，不得分配状态、驱逐仍活跃项或突破上限。
- 默认全局配额在任意十分钟窗口内最多允许初始突发 `128` 加持续恢复 `2400` 次新来源查询，因此正常最多产生 `2528` 个未过期来源状态，`4096` 项硬上限保留明确余量。该推导必须通过确定性时钟测试覆盖。
- 超出任一查询限额时统一返回 `Retry`，返回给客户端的等待时间必须限制在 `100ms` 至 `5s`；不得暴露具体命中的限速层级或内部计数。
- 不增加按 PeerId 的速率配额层；上条按 PeerId 的单并发只解决 API 归因和资源有界性，不被视为抗滥用限速。主控端使用临时 PeerId，攻击者可低成本轮换，速率保护仍由全局与来源前缀 governor 提供。
- 查询并发数、容量、速率、突发量、空闲期和重试时间必须由具有不变量的领域类型表达，不得让裸整数或无语义时长跨越配置及协议边界。所有参数允许中继操作员配置，但必须在启动时完成组合校验。
- 限速算法不得由第一方手写；已批准统一使用 `governor 0.10.4` 的 direct limiter，精确 features 和资源模型见下节及冻结依赖文档。

## 已确认的定位查询限速依赖

- 共享 core 包已批准直接依赖 `governor 0.10.4`，必须设置 `default-features = false` 且仅启用 `std` feature；禁止启用默认的 `dashmap`、`jitter` 与 `quanta` features。
- governor 只用于实现 GCRA 限速算法。全局限速器和每个已准入来源前缀分别使用 direct limiter 与单调时钟；不得使用 governor 自带的 keyed state store，因为其标准实现使用互斥锁且不能直接提供本项目要求的严格状态容量准入语义。
- 来源前缀状态表继续由 relay 网络状态机的单一所有者使用标准有界映射维护；该映射只负责容量、空闲回收和生命周期，不得重复实现 governor 已提供的 GCRA 算法。
- `std` feature 会传递启用 `futures-timer`、`futures-util`、`futures-sink` 和 `parking_lot`。当前接受该依赖成本，以直接使用 governor 的 `MonotonicClock` 并避免手写时钟适配；实际运行路径不得使用异步等待队列或带锁 keyed limiter。
- 选择该 feature 集时的独立 Windows x64 静态 CRT、Rust `1.97.0`、release 实测为：实际调用 governor 的最小二进制相对空程序增加 `13,824 B`，单个 direct limiter 值为 `40 B`，允许路径 2000 万次检查的三次结果为 `29.57ns`、`30.29ns` 与 `30.36ns` 每次。这些数据只作为依赖审批依据，完整 relay 仍须重新测量体积、分配、延迟和内存。
- governor 使用 MIT 许可证且不要求额外原生库；允许由共享 core 同时链接到 relay 和 `yon`，分别承担查询/注册滥用控制与被控端 OPAQUE 启动限速，禁止为任一处再手写限速算法。
- 已审查 governor `0.10.4` 第一方发布源码：生产源码存在一处第三方 `unsafe`，位于私有的 GCRA 参数到 `NonZeroU32` 配额重建路径，其安全前提来自内部已构造配额的不变量；本项目不得绕过公开配额构造器直接依赖该私有假设。依赖树中的原子与锁实现仍可能包含第三方 `unsafe`，必须随实际锁文件持续审计。
- governor 未声明 MSRV，这是已知缺点；当前已验证 Rust `1.97.0` 构建成功。项目采用受支持的当前稳定工具链，并必须通过 macOS、Linux musl 与 Windows 的完整目标矩阵验证静态链接和行为。
- 依赖审批时未发现 governor 及当前 feature 集的已知 RustSec 公告，但该检查不替代实际 `Cargo.lock` 上的持续 `cargo audit`、许可证审计和升级复核。

## 已确认的公开定位码分配

- 中继分配新的 20 bit 公开定位码时，必须先使用操作系统密码学安全随机源均匀生成一个 20 bit 起点，再从该起点开始按数值递增并在 `2^20` 空间内环绕，选择遇到的第一个空闲值。
- 分配必须在 relay 注册表的单一所有者任务中作为不可交错的状态转换完成，天然保证检查与插入原子性；不得为分配路径增加共享锁、跨任务 CAS 或并发随机重试。
- 默认注册容量最多为 `128`，因此只要注册表未满，环形扫描最坏最多检查 `129` 个候选并且一定成功。禁止采用固定随机重试次数后错误报告容量耗尽的概率性失败方案。
- 20 bit 空间的全部 `1,048,576` 个值均为合法定位码，包括显示为 `0000` 的值；不得保留无实际协议语义的特殊值或缩小有效空间。
- 不使用从零或固定位置开始的顺序计数器，避免中继重启后把有效定位码持续集中到可预测的小范围；也不为最多 `128` 个映射增加固定约 `128 KiB` 的全空间位图。
- 定位码在正常释放或 `Suspended` 宽限期到期后允许立即重新参与分配，不保留 tombstone。随机起点立即选回同一定位码的概率约为 `1 / 2^20`；即使发生，旧连接码的 60 bit OPAQUE 秘密也不得通过新端点认证。
- 操作系统 CSPRNG 失败时必须以结构化错误拒绝本次注册，不得退化到时间戳、进程标识、计数器、非密码学 PRNG 或其他弱随机源。
- 随机源已经统一批准为下节所述的 `rand 0.8.7`，依赖归属与全部使用边界由冻结依赖文档确定。

## 已确认的统一安全随机源

- 项目统一复用已批准的兼容依赖 `rand 0.8.7` 作为操作系统 CSPRNG 入口，不新增 `getrandom` 直接依赖。该依赖由提供安全随机抽象的最小共享包声明，使 `yon` 与 relay 复用同一实现；不得因方便而在多个 workspace 包重复声明不同随机库或主版本。
- 随机能力必须通过职责单一的 `SecureRandom` trait 表达，最小能力面为尝试填充调用方提供的可变字节切片并返回结构化错误。生产实现包装无持久内部状态的 `rand::rngs::OsRng`，调用方使用静态分发；不得使用 trait object、全局可变 RNG、共享锁或隐藏的线程局部伪随机状态。
- 第一方生产实现只允许调用可传播失败的 `try_fill_bytes`；禁止第一方调用会把操作系统随机源故障转换为 panic 的 `fill_bytes`，也禁止在失败后退化到时间、计数器、进程信息或非密码学 PRNG。已接受 `opaque-ke 4.0.1` 内部因其公开 RNG trait 限制直接调用 `fill_bytes` 的精确第三方残余风险，不得扩大到其他第一方调用或弱随机 fallback。
- 连接码秘密、身份密钥字节、认证 nonce 和中继公开定位码随机起点均通过该统一能力取得系统熵；各领域类型仍须在自己的边界完成长度、掩码、不变量与秘密生命周期管理。`opaque-ke` 因公开 API 强制要求 Rand 0.8 `RngCore`，由 Pake 适配层把同一 `OsRng` 直接交给该第三方 crate，不实现会把错误转成 panic 的第一方 `RngCore` 适配器；该边界受上一条精确第三方例外约束。
- `SecureRandom` 填充已有缓冲区的生产路径不得产生堆分配；确定性测试实现必须显式注入且只存在于测试路径，禁止成为生产失败时的 fallback。
- 独立 Windows x64 静态 CRT、Rust `1.97.0`、release 测量中，实际调用 `rand 0.8.7` 的 `OsRng` 与直接调用最新 `getrandom 0.4.3` 的最小二进制均相对空程序增加 `1,536 B`。当前选择复用 Rand 0.8 是为了避免增加第二个直接随机源依赖，而不是基于未证明的性能差异；完整产物仍须跨平台复测。
- `rand 0.8.7` 及其系统随机传递依赖内部包含第三方 `unsafe` 与平台接口代码，必须持续执行实际锁文件的安全审计；本批准不允许第一方代码出现任何 `unsafe` 或自定义随机后端。

## 已确认的 libp2p 应用子流依赖

- 已批准直接依赖 rust-libp2p 官方维护的 `libp2p-stream 0.4.0-alpha`，必须设置 `default-features = false`；该版本没有可选 feature，不得为其声明无意义的 feature。
- 该依赖用于在已建立并完成传输层身份认证的 libp2p 连接上协商和承载项目自有应用协议子流，避免第一方手写 `ConnectionHandler`、升级协商器和相关底层状态机。
- 项目所有者已明确接受该依赖当前仍为 alpha 预发布版本。这是仅针对 `libp2p-stream` 的受限例外，不构成允许其他依赖使用预发布版、Git 版本或未发布补丁的通用授权。
- `libp2p-stream` 的不稳定公开 API 不得越过共享网络包边界；必须通过项目自己的最小 trait 和类型安全错误模型隔离，以便其稳定版或更合适实现发布后替换。
- 该依赖在接收方未及时轮询时可能丢弃新入站子流，因此每个协议的接收循环必须持续排空并把子流交给有硬上限的会话状态机；不得用无界队列掩盖背压。该行为必须有饱和、取消、关闭和慢消费者测试。
- 每次升级必须重新核对其与已批准 rust-libp2p 版本的兼容性、预发布状态、MSRV、依赖树、安全公告、背压语义、跨平台构建和资源测量；一旦发布可用稳定版，必须优先重新评审并退出本例外。

## 已确认的 rust-libp2p 依赖与临时 DNS 安全例外

- 已批准直接依赖当前最新稳定版 `libp2p 0.56.0`，必须设置 `default-features = false`，仅启用 `autonat`、`dcutr`、`dns`、`ed25519`、`identify`、`macros`、`memory-connection-limits`、`noise`、`ping`、`quic`、`relay`、`tcp`、`tokio`、`upnp`、`websocket`、`yamux` features。
- 项目所有者已明确决定不等待下一版 rust-libp2p 发布，临时接受 `libp2p 0.56.0` 的 `dns` feature 经 `libp2p-dns 0.44.0` 引入 `hickory-proto 0.25.2`，以及该精确版本命中的 `RUSTSEC-2026-0118`、`RUSTSEC-2026-0119` 两项拒绝服务公告。
- 上述决定只豁免这两个精确公告和该条已审查依赖路径；另有独立批准的 `RUSTSEC-2024-0436` `paste 1.0.15` 构建期停止维护例外，精确路径和退出条件见冻结安全文档。除此之外不允许忽略其他漏洞、soundness 问题或维护警告。CI 必须逐项列出三个编号，禁止忽略全部公告、整个 crate 或严重级别。
- `RUSTSEC-2026-0118` 只在启用 Hickory `dnssec-ring` 或 `dnssec-aws-lc-rs` 并进行 DNSSEC 验证时可达；当前实际 feature graph 只允许 `std`、`system-config`、`tokio` 相关路径，必须持续断言没有启用任何 Hickory DNSSEC、TLS、HTTPS、QUIC 或 H3 feature。feature graph 发生变化时，本例外立即失效并须重新审批。
- `RUSTSEC-2026-0119` 位于 Hickory DNS 消息编码器。Yonder 只允许解析由本机操作员在启动配置中提供并经过长度与语法校验的普通 `/dns4`、`/dns6` 主机名，不运行 DNS 服务器、递归器或代理，不允许网络对端通过 Yonder 应用协议提交待编码 DNS 记录。v1 禁止 `/dnsaddr`，避免把不可信 TXT 记录递归扩展为额外 Multiaddr 与查询。
- DNS 解析必须受连接总超时、候选数量和并发上限约束；当前复用 `libp2p-dns 0.44.0` 的单次最多 `16` 个 dial attempt、`32` 个 DNS lookup 硬限制，并叠加冻结设计的 8 个操作员 relay 地址、同目标 8 个已建立候选和每 transport 2 个在途限制。解析失败只能使相应候选失败，不得阻塞 Swarm、产生无界重试、无界记录集合或无界内存增长；升级必须复核上游常量，所有限制必须有恶意或异常 DNS 响应、超时、超量地址与取消测试。
- 每次构建必须验证锁文件中该路径仍为 `libp2p 0.56.0 -> libp2p-dns 0.44.0 -> hickory-resolver 0.25.2 -> hickory-proto 0.25.2`。任一版本、依赖路径、feature 或公告内容变化都必须停止沿用本例外并重新评审。
- rust-libp2p 一旦发布兼容的稳定版本并改用不受这两项公告影响的 Hickory 版本，必须立即提交依赖升级方案；完成跨平台、互操作、性能和安全回归后删除公告忽略项及本临时例外，禁止仅为避免升级工作而继续固定旧版。

## 最终审核补充决策

- 已批准把直接依赖更新为当前稳定版 `futures 0.3.33`、`serde 1.0.229`、`thiserror 2.0.19`、`tokio 1.53.0`；各自继续禁用默认特性并保持冻结 feature 集不变。四者许可证与 MSRV 均兼容现有基线，升级后必须重跑完整依赖、安全、跨平台、性能和产物验证。
- 连接码位置参数必须在进入第一方所有权时立即放入可清除容器；Clap 只做无失败接收，领域解析在应用边界执行。任何无效连接码错误、Debug、日志或测试输出均不得包含原始位置参数；Clap 为满足其 `Clone` 边界所需的共享必须使用不复制秘密字节的共享所有权，并在移交领域解析时验证唯一所有权。
- OPAQUE `ServerFinish` 完成后，host 网络 owner 必须先重验唯一连接并同步提交 `Authenticating -> AwaitingTerminal`，再发送和 flush `Authenticated`。确认阶段必须持续驱动 Swarm，并以有界 `PendingPair` 保存目标 PeerId 每类第一条 terminal 子流；禁止在确认 I/O 尚未结束时丢弃合法子流。确认 flush 后启动 `10s` terminal 绝对截止。
- 产品 stdout 不得使用遇到 broken pipe 会 panic 的 `print!`/`println!`。连接码、relay identity 和 public external 地址必须通过可测试的 `Write` 边界写入、flush 并传播结构化 I/O 错误。连接码输出失败时必须尽力立即释放刚分配的 locator；relay stdout 只输出 `external + PeerId` 和 PeerId 行，listen 地址只进入诊断日志。
- Unix `SHELL` 必须是无 NUL 的绝对路径并指向现存可执行普通文件，否则回退 `/bin/sh`；Windows `COMSPEC` 必须是指向现存普通文件的绝对路径，否则回退 `cmd.exe`。ConPTY 无法在不拆除 pseudoconsole 的前提下表达通用输入 EOF，因此 Windows 非交互脚本必须显式发送 shell 退出命令，不得用可能截断输出的伪 EOF 掩盖该平台边界。
- Windows `PROGRAMDATA` 缺失、非 Unicode、为空或非绝对路径时继续 fail-closed；不为绕过系统层定位错误而猜目录或引入不安全 FFI。README 和产品契约必须明确此前置条件。
- 仓库采用 `LICENSE-MIT` 与 `LICENSE-APACHE` 双许可证文本。已批准发布工具 `cargo-about 0.9.1`，仅启用 `cli`，用于生成独立 `THIRD-PARTY-LICENSES.html`；许可证资产必须纳入 checksum 和 provenance，但不得放入单二进制归档。
- 在性能、资源、长时 fuzz、完整网络/恶意 relay、五目标覆盖率和六目标原生发布证据全部接入前，tag workflow 只允许生成并证明 release candidate artifact，不得自动创建正式 GitHub Release。MSRV 门禁必须在六个原生目标上实际 build/link 生产 workspace，不能只运行 `cargo check`。
